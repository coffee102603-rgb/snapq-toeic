📘 SnapQ 사전·사후 설문 설계 및 운영 규칙 (최종 확정본)
※ 개발서 / dev_log_snapq.txt 보관용
※ Q4 학습 시간 문항: 고강도 학습자(5시간 이상) 반영
========================================================

[0] 설계 목적 및 기본 원칙
- 본 설문은 SnapQ 플랫폼을 활용한 논문 연구 목적의 필수 데이터 수집 장치이다.
- 학생 입력은 최소화하고, 설문은 각 2분 내 완료 가능하도록 설계한다.
- 실명 및 개인정보는 수집하지 않으며, 닉네임 기반 가명 ID만 사용한다.
- 사전·사후 설문은 동일 문항을 사용하여 변화량(pre-post difference)을 분석한다.
- 설문 설계로 인해 플랫폼 운영이 무거워지지 않도록 한다.

--------------------------------------------------------

[1] 사전 설문(Pretest) 설계 – 첫 접속 강제 게이트

■ 노출 조건
- 해당 월 첫 접속 시 반드시 1회 노출
- pretest_done == False 인 경우에만 표시

■ 접근 제어
- 사전 설문 응답 전까지:
  · 메인 허브 접근 불가
  · 모든 전장(P5 / P7) 접근 불가
- 설문 화면만 단독으로 노출

■ 완료 처리
- 설문 제출 성공 시:
  · pretest_done = True
  · pretest_ts 저장
  · 즉시 플랫폼 정상 접속 허용
- 이후 재접속 시 사전 설문 재노출 없음

--------------------------------------------------------

[2] 개인 시작일(first_login_date) 정의
- 학생별 개인 시작일 = SnapQ 최초 접속일(first_login_date)
- 늦게 등록한 학생도 충분한 학습 노출 기간을 보장하기 위한 기준
- 사후 설문 및 연구 기간 계산의 기준 날짜로 사용

--------------------------------------------------------

[3] 사후 설문(Posttest) 설계 – 기간 기반 강제 + 연구 제외 플래그

■ 사후 설문 응답 윈도우 (개인 기준, 포함 계산)
- 개인 시작일 D 기준:
  · 사후 설문 강제 시작: D + 20일
  · 사후 설문 종료: D + 25일
- 예:
  · D = 1월 5일
  · D+20 = 1월 25일 (강제 진입 시작)
  · D+25 = 1월 30일 (응답 윈도우 종료)

■ 접근 제어 (응답 윈도우 내)
- 오늘 날짜 ≥ D+20 AND posttest_done == False 인 경우:
  · 메인 허브 접근 잠금
  · 모든 전장 접근 잠금
  · 사후 설문 화면만 노출
- 설문 제출 즉시:
  · posttest_done = True
  · posttest_ts 저장
  · 잠금 해제 및 정상 이용 가능
- 설문은 1회만 응답하며, 완료 후 재노출되지 않음

■ 응답 윈도우 종료 후 (D+25 초과)
- 사후 설문 미응답자도 플랫폼 접속은 허용
- 대신 연구 제외 플래그 부여:
  · posttest_missed = True
  · research_excluded = True
- 논문 분석 시 해당 학생 데이터는 제외 처리

--------------------------------------------------------

[4] 사전·사후 설문 문항 (동일 문항 사용)

[Ⅰ. 배경 정보 문항] (객관식)

Q1. 현재 본인의 TOEIC 점수 수준은?
- 600점 미만 / 600~699 / 700~799 / 800~899 / 900 이상 / 최근 점수 없음

Q2. 이번 학습을 통해 목표로 하는 TOEIC 점수는?
- 700 / 750 / 800 / 850 / 900 이상

Q3. 목표 시험 시기는 언제인가요?
- 1개월 이내 / 2~3개월 이내 / 4~6개월 이내 / 아직 미정

Q4. 최근 기준, 하루 평균 TOEIC 학습 시간은?
- 거의 하지 않음
- 1시간 미만
- 1~2시간
- 2~3시간
- 3~5시간
- 5시간 이상

--------------------------------------------------------

[Ⅱ. 자기 인식 문항] (리커트 5점 척도)
※ 1 = 전혀 그렇지 않다 / 5 = 매우 그렇다

Q5. 시간 제한이 있으면 오히려 집중이 더 잘 된다.
Q6. RC 문제를 풀 때, 감이 아니라 이해한 내용을 기준으로 판단한다.
Q7. 문제를 푸는 동안 잡생각이 줄고, 문제에 몰입하는 편이다.

--------------------------------------------------------

[5] 저장 필드 요약 (개발 기준)

- first_login_date (date)

- pretest_done (bool)
- pretest_ts (datetime)

- posttest_done (bool)
- posttest_ts (datetime)
- posttest_missed (bool)
- research_excluded (bool)

--------------------------------------------------------

[6] 운영 안전장치 (필수 구현)

- 설문 저장 성공 시 명확한 성공 메시지 표시
- 중복 제출 방지 (done=True 이후 설문 폼 완전 비노출)
- 파일/폴더 미존재 시 자동 생성 및 예외 처리
- 교사용 비상 우회 코드 1개 제공 (운영 안전용, 평시 미사용)

--------------------------------------------------------

[7] 연구 활용 관점 요약

- 사전·사후 동일 문항 사용 → 인식 변화량 분석 가능
- 개인 시작일 기준 설계 → 늦은 등록자도 공정한 분석 가능
- 고강도 학습자(5시간 이상) 구간 분리 → 천장 효과 방지
- 사후 미응답자 연구 제외 → 데이터 질 유지
- 강제 게이트 설계 → 사전 데이터 누락 방지

--------------------------------------------------------

[8] 핵심 설계 선언 (최종)

- 사전 설문은 “플랫폼 첫 접속의 필수 조건”이다.
- 사후 설문은 “충분한 학습 노출 이후(D+20~25)”에 강제한다.
- 사후 설문 미응답자는 연구에서는 제외하되, 플랫폼 사용은 허용한다.
- SnapQ는 연구 신뢰성과 운영 안정성을 동시에 확보하는 구조를 채택한다.

- 허브 진입 시:

- 월 변경 시 자동 차단:

==============================
[SnapQ Dev Log]
Phase 2 — P7 전투 UI 규격 고정 (DESIGN STANDARD v1.0)
==============================

■ 목적
P7 Reading Arena 전투 화면을 SnapQ의 ‘표준 전장 UI’로 확정한다.
이 규격은 이후 P5, Vocab Timebomb, 기타 전투형 모듈의 기준이 된다.

※ 본 단계에서는 전투 로직, 데이터 구조, 타이머, 점수 계산을 절대 변경하지 않는다.
※ UI 배치/네이밍/규격만 정의한다.

--------------------------------
1. 기본 원칙 (ABSOLUTE RULES)
--------------------------------
- 전투 로직 0% 변경
- HUD, 버튼, 라벨은 항상 같은 위치 유지
- 화면은 게임 HUD처럼 ‘익숙함’을 최우선
- 전투 화면은 아래 3구획만 기억하면 되도록 설계

  [MISSION] / [INTEL] / [COMBAT]

--------------------------------
2. 네이밍 표준 (고정 문구)
--------------------------------

[상단 타이틀]
- 🔥 P7 Reading Arena

[우측 상태 표시]
- 랭크 LvX · 연승 Y/5 · 다음 레벨

[미션 바 (항상 동일)]
- 🔥 미션: 핵심만 잡아라
- 🎯 룰: 3연속 명중 = 연승 유지
- ⏱ 폭탄: 시간 끝나면 실패

※ 위 3줄은 P7 브랜드 문장으로 고정한다.

--------------------------------
3. 전투 화면 레이아웃 규격
--------------------------------

[기본 레이아웃]
- 2열 고정 구조

┌───────────────┬─────────────────────┐
│ 🛰️ INTEL       │ 🎯 COMBAT             │
│ (지문)         │ (문제 / 선택지 / HUD) │
└───────────────┴─────────────────────┘

--------------------------------
4. INTEL 패널 (왼쪽)
--------------------------------
- 섹션명: 🛰️ INTEL
- 영어 지문 표시
- 해석은 접힘/펼침(toggle) 방식으로만 제공
- 전투 중 해석은 기본적으로 접힌 상태 유지

--------------------------------
5. COMBAT 패널 (오른쪽)
--------------------------------
- 섹션명: 🎯 COMBAT
- 화면 상단부터 아래 순서 고정

  1) HUD (남은 시간 / 목숨 / 연승)
  2) Round 표시 (Round 1/3, 2/3, 3/3)
  3) 문제 문장
  4) 선택지 4개
  5) 제출 버튼

--------------------------------
6. HUD 규격 (위치 고정)
--------------------------------
- 위치: COMBAT 패널 최상단 고정
- 구성 요소:
  - ⏱ 남은 시간
  - ❤️ 목숨 또는 💥 폭탄
  - 연승 X/5

※ HUD는 전투 중 절대 이동하지 않는다.

--------------------------------
7. 버튼 / 인터랙션 표준
--------------------------------

[전투 중 버튼 명칭]
- 제출: 🔥 발사(제출)
- 다음 단계: ▶ 다음 라운드
- 후퇴/포기(있을 경우): ↩ 후퇴

[선택지 표기 규칙]
- A) B) C) D) 고정
- 선택지는 ‘무기 선택’ 톤의 카드 스타일
- 카드 간 간격은 촘촘하게 유지

--------------------------------
8. 컬러 및 강조 규칙
--------------------------------
- 미션 바: 오렌지 포인트 컬러 유지
- 정답: 초록(✅)
- 오답: 빨강(❌)
- 전투 중 불필요한 여백 최소화

--------------------------------
9. Phase 2 완료 조건 (CHECKLIST)

✅ [P7 HUD 표준안] Phase 3 — HUD Visual Spec (A-1 → A-3 → A-2)

[0) 절대 원칙]
- 로직 무변경: 타이머/콤보/연승/정답판정/session_state/저장 로직은 절대 건드리지 않는다.
- UI만 변경: HUD 영역(우측 계기판)의 “보이는 것(스타일/레이아웃/강조)”만 수정한다.
- 주인공 우선: 지문/문제/선택지가 메인. HUD는 보조 계기판이며 화면을 뺏지 않는다.
- 일관성: P7에서 확정된 HUD 톤/레이아웃이 이후 P5/Vocab/Armory에도 표준으로 확장된다.

[1) A-1 (확정) — FPS 계기판 HUD (Neon Gauge / High Contrast)]
(목표)
- 우측 HUD가 “게임 계기판”처럼 보이게 만든다.
- 정보는 그대로 유지하되 가독성/고정감/긴장감만 상승시킨다.

(적용 범위)
- 전투 화면 우측 HUD 구성요소만:
  1) 랭크/연승 카드
  2) TIMEBOMB 카드
  3) COMBO 카드
  4) HUD 버튼(병기고 이동 등)

(스타일 규칙)
- HUD 카드 공통:
  - 좌측 border 포인트 바(오렌지 계열)
  - 얕은 그라데이션 배경 + soft shadow(붙어있는 패널 느낌)
  - 숫자/핵심 지표는 굵게/크게(한눈에 KPI)
- TIMEBOMB:
  - 남은 시간은 게이지/바로 “감소 체감”이 나게 강조
- COMBO:
  - 연속 성공 지표를 배지/굵은 수치로 “즉시 인지”되게 배치(문구 최소화)

(금지)
- st_autorefresh 키/버킷/틱 로직 변경 금지
- 타이머 계산 방식 변경 금지
- session_state 키 변경/통합 금지

(완료 기준)
- 한눈에 “RANK / TIMEBOMB / COMBO”가 읽힌다.
- 메인(지문/문제/선택지) 집중을 방해하지 않는다.
- 기존 기능(타이머/콤보/병기고 이동/전투 흐름)은 동일 동작한다.

[2) A-3 (다음) — PC방 게임 HUD 강조 연출 (Light Animation / Alert Only)]
(목표)
- A-1을 유지한 채 “상태 변화 순간”에만 반응한다.
- 과한 애니메이션 금지, 최소 연출만 허용한다.

(적용 범위: UI-only)
- TIMEBOMB: 위험 구간(예: 30% 이하)에서만 약한 pulse/glow
- COMBO: 콤보 상승 순간에만 가벼운 강조(색/배지/아이콘)

(규칙)
- 애니메이션은 짧고 약하게(항상 ON 금지)
- 시각적 소음/피로가 생기면 즉시 롤백

(완료 기준)
- 시간 부족/연승 유지의 긴장감이 즉각 체감된다.
- 눈이 피로하지 않고 메인 콘텐츠 방해가 없다.

[3) A-2 (마지막) — 시험 실전 HUD 스킨 (Clean / Minimal)]
(목표)
- 같은 HUD 구조(레이아웃)는 유지하고 “실전 모드 톤”을 정의한다.
- 지금은 토글 구현 없이 스킨 정의만 확정한다(토글은 추후).

(적용 범위)
- A-1 레이아웃 유지
- 색/그라데이션/포인트를 절제해 “시험장 UI”로 전환 가능하게 설계

(스타일 규칙)
- 대비는 유지하되 색은 최소화
- 정보 밀도 ↑, 불필요한 아이콘/문구 최소화

(완료 기준)
- 구조/로직 변경 없이도 “게임 모드(A-1/A-3)” ↔ “실전 모드(A-2)” 톤 전환이 가능해진다.

[4) 적용 순서(확정)]
1) A-1 적용
   - 기본 HUD 톤/레이아웃 확정
   - RANK / TIMEBOMB / COMBO 시각 계기판화
   - 모든 수치는 기존 로직 그대로 유지

2) A-3 추가
   - 상태 변화 시에만 제한적 시각 강조

   - 상시 애니메이션 금지

3) A-2 스킨 정의
   - HUD 구조/위치는 그대로 유지
   - 색감·아이콘·강조를 절제한 “실전 시험 톤” 정의
   - 토글 구현은 Phase 4 이후로 유보

[5) Phase 3 완료 선언]
- P7 HUD 레이아웃은 본 문서를 기준으로 “동결(freeze)”한다.

  ❌ HUD 위치 이동

  ❌ 공통 CSS 대수술

   - Vocab Timebomb 세트는 항상 prepare_vocab_quiz_set()을 통해 초기화할 것
   - 추후 디자인만 손봐도 되지만, 핵심 로직(목숨 3개, 오답 시 감소, 0이면 허브 복귀)은 그대로 유지
